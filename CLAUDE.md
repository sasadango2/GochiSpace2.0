# GochiSpace

美味しい発見をシェアする、飲食店レビュー共有サービス（再構築版）

## プロジェクト概要
家族・友人など信頼できる人同士でのみ飲食店レビューを共有できるWebサービス。
匿名性・サクラ投稿の問題を解消し、相手の好みを理解した上で飲食店を選べる環境を実現する。

## 主な機能
- ユーザー認証（Supabase Auth）
- 相互フォロー機能（クローズドな設計）
- マップ表示機能（地図上のマーカーからレビューを閲覧）
- 嗜好設定（好きなジャンル最大3つを登録）
- フォローユーザー検索・フォロー申請
- レビュー投稿（店舗検索・カテゴリー・評価・コメント）

## 技術スタック

### フロントエンド
- **React 19** + **TypeScript** + **Vite**
- **React Router v7** - ルーティング
- **Material-UI (MUI) v7** - UIコンポーネント
- **Leaflet.js + OpenStreetMap** - 地図表示（完全無料）
- **Google Places API（New）** - 店舗検索（Text Search・無料枠内で運用）

### バックエンド（Docker で管理）
- **Node.js + Hono + TypeScript**
- Docker / Docker Compose でローカル開発・デプロイ管理

### データベース・認証
- **Supabase**（PostgreSQL）- データベース（無料枠：500MB）
- **Supabase Auth** - ユーザー認証

### ホスティング
- フロントエンド：**Vercel**（無料）
- バックエンド：**Render** または **Railway**（無料枠）
- DB：**Supabase**（無料枠）

## コスト方針
収益がインフラ費を上回るまでは完全無料枠での運用を続ける。有料APIは使用しない。

## 収益化方針

### 基本原則
- **信頼性を損なう収益化はしない**。店舗からの広告掲載・レビューの優先表示は、サクラ排除というサービスの存在意義を壊すため行わない
- ユーザーのレビューデータの外部販売はしない
- 無料ユーザーの既存機能は制限しない（後から機能を取り上げない）

### 段階的ロードマップ

#### Phase 1: ユーザー基盤の拡大（現在）
- 全機能無料で提供し、相互フォローによる招待経由の成長を優先する
- 収益化はまだ行わない。無料枠内でのコスト管理を徹底する

#### Phase 2: アフィリエイト（送客手数料）
- レビュー詳細・行きたいリストから予約サイトへの送客リンクを設置する
- 候補：ホットペッパーグルメ / 食べログ / 一休.com のアフィリエイトプログラム（バリューコマース・A8.net 経由）
- 広告と異なりユーザーの行動（予約したい）と一致するため、UXと信頼性を損なわない

#### Phase 3: フリーミアム（プレミアムプラン）
- 決済は **Stripe**（サブスクリプション）を想定
- プレミアム機能の候補：
  - レビュー写真の上限拡張（3枚 → 10枚）
  - 嗜好ジャンルの登録数拡張（3つ → 無制限）
  - グループ機能（家族・友人グループ単位での共有）
  - レビューデータのエクスポート

### 技術的な準備事項
- `profiles` にプラン情報（または `subscriptions` テーブル）を追加し、SQLで機能制限を判定する
- Stripe Webhook を受けるエンドポイント（`POST /webhooks/stripe`）をバックエンドに追加する
- 機能制限はフロントの出し分けだけでなく、必ずバックエンドでも検証する
- 有料化前に特定商取引法に基づく表記・プライバシーポリシー・利用規約の整備が必要

## コーディング規約
- 変数・関数名：camelCase
- 関数は動詞始まり（例：fetchRestaurant, handleReview, validateUser）
- 1関数50行以内
- TypeScript の型定義を必ず書く（any 禁止）
- コメントは必要最低限（WHYが非自明な場合のみ）
- スマホとPCでの操作を想定したUIにすること

## DB設計方針
- PostgreSQL（リレーショナル）で相互フォローをSQLのJOINで管理
- Firestoreのような非正規化は行わない

## 今後の実装候補（優先度順）

### 透明化系（コンセプト直結・優先度高）
「信頼できる人のレビュー」という価値を強めるため、「誰が・どんな好みで・どう評価する人か」を可視化する。

1. **フォロワーのプロフィール透明化**
   - ユーザー別レビュー一覧（`/follows/:userId`）の上部に嗜好ジャンル・レビュー件数を表示する
   - 相互フォロワー限定のプロフィール取得APIをバックエンドに追加する（現状 `/users/me/genres` は自分専用）
2. **評価傾向の可視化（辛口・甘口メーター）**
   - ユーザーごとの評価分布（また行きたい/普通/好みじゃなかった の割合）を表示する
   - 既存レビューデータの集計のみで実現可能
3. **フィード・マップの「人」フィルタ**
   - 特定のフォロワーのレビューだけに絞って表示できるようにする

### 体験改善系（中優先度）
4. **通知の既読処理** — `unread-count` APIはあるが既読化APIが見当たらない。バッジが消えない可能性があるため実機確認のうえ対応する
5. **PWA対応** — manifest + service worker でホーム画面追加に対応する（無料）

### 設定まわりの残タスク
- **メール認証URLのlocalhost問題** — Supabaseダッシュボードの Site URL を本番URLに変更し、コード側でも `signUp` に `emailRedirectTo: window.location.origin` を指定する
