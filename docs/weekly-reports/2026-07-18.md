## Project 概要
- **GochiSpace**：家族や友人、知人など互いに知り合い同士の関係のみでユーザーがレストランのレビューを投稿・共有できるサービス。
- **なぜ**：既存の主要な飲食店レビューサイトは匿名性が高く、サクラレビューや個人の主観によるノイズが混在しており、情報の信頼性に欠けるという課題がある。身近で信頼できる人物の間で安心して飲食店を選べる仕組みが必要だと感じたため。
- **どこに向かって**：相互フォローによるクローズドな関係性のなかで完全に信頼できるレビューのみを共有し、味の好みや店舗の雰囲気を把握した上で、不安なく最適な飲食店選定ができる新たな体験の提供を目指す。

## 課題 / 相談事
- 透明化機能の設計で「少数データの統計表示は誤解を生む」という問題に直面しました。クローズドなサービスは母数が小さく、レビュー3件で「また行きたい75%」と出すのは誠実ではないため、10件未満は割合ではなく生の件数のみを表示する方針としました。
- 同様に「辛口・甘口」のようなラベリングは、家族・友人のクローズドな場では投稿者の萎縮につながるリスクがあると判断し、分布はそのまま見せて解釈は読み手に委ねる（ラベルを付けない）方針にしています。この匙加減が適切かはユーザーが増えた段階で検証したいと考えています。
- スマホのロック画面へのプッシュ通知は Web Push（PWA）で無料で実現可能なことを確認済みですが、iOS は「ホーム画面に追加」した PWA からのみ動作するため、PWA対応を先行させる必要があり、いったん保留としました。

## 学び / シェアしたい情報
- 📝 通知バッジの設計で「未読」と「要対応」は別物だと学びました。バッジを pending 件数と連動させると申請を放置した場合に永久に消えず、通知自体が信用されなくなります。バッジは「最後に開いた日時より後に届いたもの」だけを数え、対応待ちは一覧に残す分離設計にしました。
- 📝 ローカルの DB 初期化 SQL と本番 DB のスキーマがいつの間にか乖離していました（本番にだけ存在するテーブル・カラムが複数）。集計機能の実装前に同期を行い、以後はマイグレーション SQL をリポジトリに残す運用に変えました。「本番を先にダッシュボードでいじると必ずズレる」という教訓です。
